### Context
운영 환경에서 발생하는 장애를 신속하게 파악하고 시스템의 상태를 모니터링하기 위해서는 체계적인 로그 전략(Logging Strategy)이 필수적입니다. 단순히 텍스트를 기록하는 것을 넘어, 분석 가능한 데이터 형태의 로그 설계 방안을 정리합니다.

### Core
* 로그 레벨 기준
    * DEBUG: 개발 환경에서 세부적인 로직 흐름 확인용
    * INFO: 비즈니스 핵심 이벤트 및 서비스 상태 변화 기록
    * WARN: 잠재적 오류가 예상되나 시스템이 복구 가능한 상황
    * ERROR: 즉각적인 조치가 필요한 심각한 예외 발생 상황
* 구조화 로그(Structured Logging)
    * 사람뿐만 아니라 기계가 읽기 쉬운 JSON 형식으로 로그를 출력합니다.
    * 필드 기반의 검색과 집계(Aggreagation)가 용이해집니다.
    * 예시: `{"timestamp": "2026-03-10T10:00:00Z", "level": "INFO", "message": "Order processed", "orderId": "12345"}`
* Trace ID 활용
    * 요청(Request)의 전체 생명주기를 추적하기 위해 진입점에서 유니크한 `traceId`를 생성하고 헤더를 통해 서비스 간에 전파합니다.
    * 분산 시스템에서 특정 트랜잭션의 흐름을 단번에 시각화할 수 있습니다.
* 민감정보 마스킹(Masking)
    * 개인정보보호법 준수를 위해 PII(개인식별정보)를 로그에서 제거하거나 마스킹 처리합니다.
    * 정규식(Regex)을 이용한 필터링 레이어를 애플리케이션 로거 혹은 로그 수집기단에 적용합니다.
* 중앙 로그 수집(Centralized Logging)
    * 모든 서버의 로그를 단일 저장소로 수집하여 관리합니다.
    * ELK Stack(Elasticsearch, Logstash, Kibana) 또는 Grafana Loki를 통해 시각화와 실시간 알림을 설정합니다.

### Insight
구조화된 로그와 Trace ID의 결합은 현대적인 분산 아키텍처에서 디버깅 시간을 획기적으로 단축해 줍니다. 특히 서비스가 확장될수록 파편화된 로그를 하나로 묶어주는 Trace ID는 없어서는 안 될 도구입니다. 단순히 로그를 쌓는 것에 그치지 않고, 수집된 로그를 바탕으로 비정상적인 패턴을 탐지하는 모니터링 시스템까지 구축하는 것이 로그 전략의 최종 목표입니다.
