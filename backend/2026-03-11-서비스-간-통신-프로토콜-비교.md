### Context
마이크로서비스 아키텍처(MSA)를 설계할 때, 서비스 간 통신(Inter-service communication) 방식을 선택하는 것은 시스템의 성능, 확장성 및 개발 생산성에 직결됩니다. REST, GraphQL, gRPC는 각기 다른 프로토콜과 철학을 바탕으로 하며, 요구사항에 따라 적절한 선택이 필요합니다.

### Core

* RESTful API: HTTP/1.1 프로토콜 기반의 리소스 중심 API입니다. 범용성이 높고 구현이 단순하여 웹 생태계에서 표준으로 자리 잡았습니다. JSON을 데이터 형식으로 사용하며 상태를 유지하지 않는(Stateless) 구조를 가집니다.
* GraphQL: 클라이언트가 필요한 데이터 구조를 직접 정의하여 쿼리하는 API 쿼리 언어입니다. 오버페칭(Over-fetching)과 언더페칭(Under-fetching) 문제를 해결하며, 단일 엔드포인트를 통해 유연한 데이터 조회가 가능합니다.
* gRPC: Google에서 개발한 고성능 RPC(Remote Procedure Call) 프레임워크입니다. HTTP/2를 기반으로 하며, 프로토콜 버퍼(Protocol Buffers, Protobuf)를 사용하여 직렬화합니다. 이진 데이터(Binary)를 전송하므로 REST보다 훨씬 빠르고 효율적입니다.

서비스 간 통신 비교 예시:

* gRPC (내부 통신): 고성능 및 강력한 타입 안정성이 필요한 서비스 간 백엔드 호출에 적합합니다.
* REST (퍼블릭 API): 외부 개발자나 클라이언트가 쉽게 접근해야 하는 공개 API 구축에 적합합니다.
* GraphQL (프론트엔드 연동): 다양한 리소스가 복잡하게 얽힌 UI 데이터를 조회할 때 적합합니다.

### Insight

기술 스택 선택 시 무조건적인 성능 우위보다는 '적합성'을 고려해야 합니다. gRPC는 강력한 타입 체크와 속도를 제공하지만 브라우저 지원이 제한적이고, REST는 환경 제약 없이 어디서든 호출 가능하지만 반복적인 요청 시 성능 오버헤드가 발생할 수 있습니다. 시스템의 성격(내부 통신 vs 외부 노출)에 따라 프로토콜을 혼합하여 사용하는 하이브리드 전략이 가장 실용적입니다.
