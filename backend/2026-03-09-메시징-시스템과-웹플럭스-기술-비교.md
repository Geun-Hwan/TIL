### Context
분산 시스템을 설계할 때 메시지 브로커(Message Broker)와 비동기 프로그래밍 프레임워크는 필수적인 요소다. 프로젝트의 요구사항에 따라 아파치 카프카(Apache Kafka), 래빗MQ(RabbitMQ), 그리고 스프링 웹플럭스(Spring WebFlux)의 적절한 선택이 필요하다.

### Core
* 아파치 카프카 (Apache Kafka)
    * 상황: 대규모 실시간 데이터 스트리밍, 로그 수집, 이벤트 소싱(Event Sourcing)에 적합하다.
    * 장점: 높은 처리량(Throughput), 영구적인 메시지 저장(Persistence), 재생 가능성(Replayability).
    * 단점: 설정이 복잡하며 운영 비용이 높다. 경량 메시지 전송에는 오버헤드가 발생할 수 있다.

* 래빗MQ (RabbitMQ)
    * 상황: 복잡한 메시지 라우팅이 필요한 경우, 전통적인 태스크 큐(Task Queue), 마이크로서비스 간 가벼운 메시징에 적합하다.
    * 장점: 다양한 메시지 라우팅 패턴 지원, 상대적으로 쉬운 운영과 설정, 낮은 지연 시간(Low Latency).
    * 단점: 데이터 처리에 특화된 카프카 대비 대규모 데이터 보관(Retention)에 제약이 있다.

* 스프링 웹플럭스 (Spring WebFlux)
    * 상황: I/O 집약적인 서비스(API 게이트웨이, 실시간 채팅, 스트리밍 서비스) 구축 시 사용한다.
    * 장점: 비차단(Non-blocking) 모델을 통해 적은 리소스로 많은 동시 연결 처리가 가능하며, 리액티브 프로그래밍을 지원한다.
    * 단점: 학습 곡선이 높으며, 블로킹 코드가 섞일 경우 성능이 급격히 저하될 위험이 있다.

### Insight
* 메시지 브로커 선택은 데이터 보존 주기와 처리량, 라우팅 복잡도에 따라 결정된다. 카프카는 '스트림' 관점에서, 래빗MQ는 '큐와 라우팅' 관점에서 접근하는 것이 좋다.
* 웹플럭스는 메시지 브로커가 아니며, 시스템의 '비동기적 상호작용'을 최적화하기 위한 서버 프레임워크이다. 고성능이 요구되는 엔드포인트에 적용해야 하며, 단순 CRUD 시스템에서는 기존 MVC 모델이 더 효율적일 수 있다.
