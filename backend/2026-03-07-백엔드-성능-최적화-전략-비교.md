### Context
백엔드 시스템의 성능 최적화를 위해 데이터 접근 전략, 캐싱, 비동기 처리, JVM 튜닝은 필수적인 요소입니다. 특히 ORM 환경에서의 N+1 문제 해결, 데이터베이스 부하 분산을 위한 Redis 도입, 고성능 처리를 위한 비동기 프로그래밍 모델의 선택 기준을 명확히 하는 것이 중요합니다.

### Core
* ORM 조회 최적화 (Lazy vs Fetch Join)
    * Lazy Loading: 지연 로딩은 연관 엔티티가 실제로 사용될 때 쿼리를 실행하여 불필요한 쿼리 발생을 방지하지만, 루프 내에서 접근 시 N+1 문제를 유발합니다.
    * Fetch Join: JPQL을 통해 연관 엔티티를 하나의 SQL 쿼리로 함께 조회합니다. N+1 문제를 해결하는 가장 대표적인 방법입니다.
    ```sql
    -- Fetch Join 예시
    SELECT m FROM Member m JOIN FETCH m.team
    ```

* 캐싱 전략 (Redis vs DB)
    * Redis: 인메모리(In-Memory) 저장소로, 읽기 집약적인 작업에서 응답 시간을 단축합니다. 데이터 일관성 처리가 중요합니다(Cache-Aside, Write-Through 패턴 등).
    * DB 직접 조회: 항상 최신 데이터를 보장하지만, 고트래픽 시 데이터베이스 커넥션 풀 고갈과 성능 저하를 유발할 수 있습니다.

* 비동기 처리 (CompletableFuture vs WebFlux)
    * CompletableFuture: Java 8 이상에서 제공하는 비동기 프로그래밍 도구입니다. 블로킹 작업을 별도 스레드 풀에서 실행하여 비동기 효과를 냅니다.
    * WebFlux: Reactor 기반의 논블로킹(Non-blocking) 라이브러리로, 적은 리소스(스레드)로 많은 요청을 처리하는 Reactive Stream 모델입니다. I/O 집약적인 서비스에 적합합니다.

### Insight
* N+1 문제는 엔티티 설계 단계부터 Fetch Join, Batch Size 설정 등을 고려하여 해결해야 합니다.
* 무조건적인 캐싱 도입보다는 '읽기 빈도'와 '데이터 변경 주기'를 고려하여 적용 범위를 결정해야 합니다.
* CompletableFuture는 전통적인 명령형 코드 유지보수에 유리하고, WebFlux는 고성능/확장성이 필요한 마이크로서비스 아키텍처(MSA) 환경에서 유리합니다. 성능 개선 시 항상 프로파일링(JVisualVM, Micrometer)을 병행하여 병목 지점을 데이터로 증명해야 합니다.
